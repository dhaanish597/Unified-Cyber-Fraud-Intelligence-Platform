# Authoritative Pipeline Reference

## Input contract

Every event needs `session_id` and `event_type`. SDK calls also carry
`request_id` and `correlation_id`. Missing IDs are generated by the backend.
For a given `(session_id, request_id)`, processing is idempotent: retries return
the original pipeline and decision without re-ingestion or rebroadcast.

## Stages

1. Normalize event type, timestamp, amount, IDs, and device/session identity.
2. Verify the SDK session when `require_existing_session` is true.
3. Ingest exactly once and produce `event_ack`.
4. Execute cyber threat rules from observed event fields.
5. Persist graph observations and run shared-device, beneficiary, mule,
   circular-transfer, and fraud-ring queries.
6. Execute tabular and Isolation Forest artifacts when all versioned artifacts
   are available. Otherwise return `ModelUnavailable`.
7. Select the final action in `DecisionEngineAdapter`.
8. Return the REST result and publish the same completed result to the platform
   event broker.

Phase 3 session/trust recalculation is not part of this pipeline.

## Output evidence

- Identity: `request_id`, `correlation_id`, `pipeline_id`, `session_id`.
- Delivery: `event_ack.backend_ack`.
- Dependencies: graph backend/status/GraphSAGE status; model status/version/error.
- Decision: ID, action, source, policy version, reason codes, nullable confidence.
- Timing: normalization/ingest, threat, graph, model/fallback, and total.

## Failure behavior

Validation errors return 422. Authentication/ownership errors return 401/403.
Graph query exceptions return a failed graph result without crashing the
request. Model failures return a null-score policy result. A transfer decision
is never silently queued for later financial execution; the Android UI receives
an error and may ask the user to retry.
